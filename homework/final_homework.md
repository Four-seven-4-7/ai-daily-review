## 一、 项目内容：打造自动化的复盘闭环

### 1.1 核心痛点：从“半小时搬运”到“一键洞察”
作为一个 flomo 的重度用户，我每天记录大量灵感。但过去，复盘对我来说是一件极其沉重的事情：
*   **效率极低**：我必须先手动复制当周/当月的所有散乱记录，再构思复杂的 AI Prompt 进行喂料，**平均每次搬运和调试要浪费 30 分钟以上**。
*   **框架缺失**：每天都要在脑子里现想“我该记点什么”，导致复盘质量参差不齐。
*   **解决方案**：本项目通过结构化表单规范输入，并实现一键同步与 AI 分析。现在的复盘只需点击一下，省去了所有搬运时间。

### 1.2 产品演进历程
*   **V1.0 MVP**：实现基础复盘输入与 Supabase 云端同步。
*   **V2.0 AI 洞察**：接入 ModelScope 接口，实现行为规律的深度提炼。
*   **V3.0 自动化**：上线 GitHub Actions + 飞书 Webhook 提醒工作流。

## 二、 技能增长：从“指挥官”到“架构思维”

在 8 课的练习中，我的角色发生了根本性的转变：
*   **代码的 10% 与 100%**：在本项目中，AI 实际上完成了 **100% 的代码编写**。而我作为“指挥官”，承担了剩下的 **10% 核心工作**——包括逻辑设计、格式微调以及最关键的“意图指引”。
*   **突破 Skill 的认知迷雾**：为了理解 Skill 到底是什么，我利用工作间隙查阅了大量资料。虽然目前还未在 MVP 中完全落地，但我已经为后续“冰山体系”和“道法术”系统预留了 Skill 的扩展接口。
*   **工程化自律**：经历了“需求蔓延”的惨痛教训后，我学会了严格的版本管理——实现一个功能、测试好一个版本、再进行下一次迭代。

## 三、 心路历程：关于“启动”与“修行”

### 3.1 战胜“启动难”的改点突破
最近工作强度极大，每天回家已近 9 点半。以前在这个时间点，我往往会因为“畏难情绪”陷入启动困难，瘫在沙发上刷短视频，一刷就到半夜。
**我的突破口**：我不再强迫自己按顺序看完漫长的课程视频，而是**直接从作业开始**，碰到卡壳再精准回看视频。同时，我和朋友约定在深夜会议室一起学习，这种“强制启动”的外部驱动力，让我终于拿到了新时代的船票。

### 3.2 投资小白的自我救赎
今年我原本想做投资工具，但我发现当我还不熟悉投资逻辑时，AI 无法帮我凭空创造。AI 是放大器，它只能放大你熟悉的东西。因此我回归初心，先用 AI 解决我最熟悉的“flomo 搬运”问题。

## 四、 踩坑经验：那些刻骨铭心的瞬间

### 4.1 需求蔓延的“崩溃之夜”
在做第一个 MVP 作业时，由于觉得“用得不舒服”，我不停地向 AI 下达新需求（如本地缓存）。结果导致状态同步冲突，整个项目几乎崩盘。
*   **决断时刻**：面对几乎报废的代码，我意识到重构太慢，果断指挥 AI **“止损回退”**到之前的稳定版本。
*   **感悟**：不要在开发过程中随意动摇。先跑通，再优化。

### 4.2 GitHub 惊魂：密钥泄露的拦截
在一次上传代码后，我收到了 GitHub 的敏感信息警告邮件，心跳瞬间加速，非常紧张。
*   **反思**：虽然我信任 AI 编码，但作为指挥官，我必须对敏感数据保持绝对的谨慎。这次“泄露危机”让我对 API 安全管理有了实战级的认知。

## 五、 未来规划：构建“冰山”与“道法术”

目前的版本只是冰山一角，我计划在未来加入两个核心体系：

### 5.1 冰山体系：纯粹的情绪觉察
我习惯记录那些与自我的关系、幸福的瞬间。我不需要 AI 的安抚或建议，我只需要它**通过 Skill 识别并精准记录**我的情绪快照，构建我个人的情绪档案。

### 5.2 道法术：理性的智慧沉淀
*   **案例**：针对“决策纠结”，我的**术**是“设定决策截止时间（Deadline）”。
*   **AI 辅助**：我会让 AI 帮我剖析背后的**道**（能耗陷阱、无解内耗）与**法**（快速翻篇的方法论），从而形成一套可操作的智慧闭环。
