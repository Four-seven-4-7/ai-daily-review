## 一、 项目初衷：逃离半小时的“搬运循环”

作为一个 **flomo** 的重度用户，我每天记录大量灵感。但过去，复盘对我来说是一件极其沉重的事情：
*   **效率极低**：我必须先手动复制当周/当月的所有散乱记录，再构思复杂的 AI Prompt 进行喂料，**平均每次搬运和调试要浪费 30 分钟以上**。
*   **框架缺失**：每天都要在脑子里现想“我该记点什么”，导致复盘质量参差不齐。
*   **解决方案**：既然无法逃避机械的搬运，那就用代码把它彻底自动化。

## 二、 需求定义与 PRD 详述 (第 1-2 课成果)

### 2.1 痛点挖掘
*   **用户角色**：追求高效率的 flomo 知识工作者。
*   **当前做法**：`手动复制 flomo 内容` -> `打开 AI 对话框` -> `输入分析指令`。
*   **核心需求**：实现碎片记录到结构化复盘的**全自动流转**。

### 2.2 核心 PRD 内容 (数据画骨)
本项目旨在设计一个能够实现**计划自动流转**（昨日的明日变今日）和**主题深度觉察**的智能复盘工具。

#### 1. 核心功能模块
*   **今日复盘区**：自动拉取昨日规划的“明日计划”，支持 Checkbox 标记完成及备注说明。
*   **明日规划区**：输入次日核心任务。系统通过公式 `$Plan_{tomorrow}(Day_N) \rightarrow Plan_{today}(Day_{N+1})$` 实现任务自动流转。
*   **主题觉察区**：支持“主/副主题”筛选。
    *   **记忆逻辑**：系统自动默认选中用户上次提交时的主题，减少重复操作。

#### 2. 交互原型示意 (Wireframe)
```text
+------------------------------------------------------+
|  1. 今日计划完成情况 (来自昨日规划)                   |
| [x] 事项 A  [ 备注：已完成，效果不错       ]         |
| [ ] 事项 B  [ 备注：延期到明天             ]         |
+------------------------------------------------------+
|  2. 明日计划完成事项                                  |
|  - [ 输入待办任务... ]                               |
+------------------------------------------------------+
|               [    提 交 并 同 步    ]               |
+------------------------------------------------------+
```

## 三、 AI 智能接入与自动化工作流 (第 3, 7 课成果)

### 3.1 接入 LLM 洞察能力
利用 API 接入 ModelScope 的 Qwen2.5 模型，深度读取历史复盘数据，自动生成“避坑指南”。

#### AI 功能工作流说明：
1.  **数据采集**：从 Supabase 提取过去 7 天的 `todayPlans` 和 `mainThemeNote`。
2.  **Prompt 封装**：将原始数据喂给预设的“分析师提示词”。
3.  **API 调用**：通过 `fetch` 安全调用 ModelScope 接口。

## 五、 心路历程：晚九点半的深夜修行

### 5.1 投资小白的自我救赎
今年我原本想做投资工具，但我发现当我不仅是 Web 编程小白，也是投资小白时，我甚至连痛点都没有，更不知道该做什么。这段经历让我意识到：**AI 是放大器，它只能放大你熟悉的东西。** 于是我回归初心，先解决我最熟悉的“flomo 搬运”问题。这让我想起了马斯克的“五步工作法”：迭代速度，最后自动化。

### 5.2 战胜“启动难”的改点突破
最近工作非常忙，每天到家已近 9 点半。以前在这个时间点，我往往会因为畏难情绪陷入“启动困难”，瘫在沙发上刷短视频直到深夜。
**我的突破口**：不再强迫自己按顺序看完视频，而是**直接从作业开始**，碰到卡壳再精准回看。同时，我和朋友约定在深夜会议室一起学习，这种强制启动的外部驱动力，让我终于拿到了新时代的船票。

### 5.3 技能增长：从“指挥官”到“架构思维”
在本项目中，AI 实际上完成了 **100% 的代码编写**。而我承担了剩下的 **10% 核心工作**——意图指引、逻辑设计与格式微调。为了理解 **Skill** 的本质，我利用上班间隙查阅了大量资料，虽然目前还未完全落地，但已为后续功能预留了接口。

## 六、 踩坑经验：那些刻骨铭心的瞬间

### 6.1 需求蔓延的“崩溃之夜”
在做第一个 MVP 作业时，由于觉得用得不舒服，我下达了“本地临时缓存”的需求。结果导致状态同步冲突，项目几乎崩溃。
*   **感悟**：不要在开发过程中随意动摇。先跑通，再优化。最后我选择回退到稳定版本，这是我最大的收获。

### 6.2 GitHub 惊魂：密钥泄露
在一次上传代码后，收到了 GitHub 的敏感信息警告邮件，心跳瞬间加速。这让我对 API 安全管理有了实战级的认知：即使信任 AI，指挥官也必须对数据保持绝对谨慎。

## 七、 未来规划：构建“冰山”与“道法术”

现在的版本只是冰山一角，我计划在未来加入两个核心体系：

1.  **冰山体系（自我觉察）**：我习惯记录关于自身关系的瞬间。我希望未来能通过 AI 接口，在我随便输入一句话时，它就能自动辨识出这属于冰山体系中的哪个版块并归类记录。
2.  **道法术（世界道理）**：针对“决策纠结”，我的**术**是“设定决策截止时间（Deadline）”。我会让 AI 帮我剖析背后的**道**（能耗陷阱）与**法**（方法论），形成一套可操作的智慧闭环。
