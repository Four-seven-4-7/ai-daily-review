## **一、核心认知：为什么“部署”是产品的成人礼？**

有些同学可能会觉得：“我代码都写完了，部署不就是点个按钮的事吗？” 其实不然。

部署是产品的“成人礼”，因为它强制你面对真实世界的残酷：

1. **环境差异**：你的电脑环境和服务器环境不一样，本地能跑，上线报错是常态。

2. **成本意识**：云舒老师在课上花大力气讲了“算账”。在APPStore上线App要 99 美元/年的同时还需要有一台MacOS的电脑，做网站要买域名，服务器要算流量。部署，会开始让你从“写代码”的嗨点，回归到“运营成本”的冷静。

3. **交付标准**：在本地你可以随时改代码，但在云端，你需要一套机制来管理“什么版本给用户看”。

所以，这次作业不仅仅是“发个链接”，而是让你体&#x9A8C;**“我是如何把我的作品交付给世界的”**。

## **二、作业策略：条条大路通罗马**

> 原始作业说明：[ 第6课：把产品交付到真实世界——上线与发布](https://waytoagi.feishu.cn/wiki/BbtiwFvMKi3gwykmXDKcjs9InDg?fromScene=spaceOverview#OQAFdrYtVo7tIUx5Ro8cgqkUnsc)

**本次作业不限制部署平台。**

虽然课上重点讲了 **GitHub + Vercel** 这套黄金组合，但如果你的网络环境不允许，或者你更习惯其他方式（比如 Replit、Netlify，甚至是腾讯云/阿里云的静态托管），**只要评审老师能通过一个 URL 访问到你的作品，就算通过。**

### **初阶作业：让 URL 成为你的名片**

* **原始要求**：把自己开发好的网站部署到 Vercel 等其他平台/服务器上，提交部署链接。

* **我的解读**：

  * **核心目标**：打破只能在“本地运行”的壁垒。

  * **推荐路径**：就用你第五课做的那个“个人主页”或者“工具导航站”。

  * **极简操作流（GitHub + Vercel）**：

    1. **代码上云**：用 **GitHub Desktop**（云舒老师强烈推荐的可视化工具，别去死磕命令行了）把你的本地文件夹上传到 GitHub 仓库。

    2. **自动托管**：去 Vercel 官网，用 GitHub 账号登录，导入刚才那个仓库。

    3. **一键上线**：点击 Deploy，等待烟花绽放。

* **通关标准**：**<span style="color: rgb(216,57,49); background-color: inherit">你发到群里的链接，我也能点开，也能看到和你电脑上一模一样的画面，并且在作业笔记中说明你的部署步骤。</span>**

### **进阶作业：掌握“时光机”的能力**

**原始要求**：部署到 Vercel 上，并熟练使用 GitHub 发版（至少） 3 次，提交版本说明及截图。

![](images/小橙酱：第六课作业建议：让世界看见你的作品（4期第6课）-image-1.png)

**我的解读**：

* **为什么要（至少）发版 3 次？** 这不是为了凑数，而是为了让你学会使用**版本控制（Version Control）**。

* **什么是版本控制？** 它就是代码世界的“后悔药”和“时光机”。

  * **V1.0（初版上线）**：这是可能你第一次把代码推上去，可能还有点丑。

  * **V1.1（紧急修复/微调）**：比如你发现标题写错了，或者想换个背景色。在本地改完 -> 用 GitHub Desktop 提交（Commit） -> 推送（Push）。你会发现 Vercel 会自动检测到变化并重新部署。**这就是“持续集成/持续部署（CI/CD）”的雏形。**

  * **V1.2（功能迭代）**：试着加一个新的小功能，再推一次。

* **通关标准**：不仅仅是一个链接，而是&#x8981;**<span style="color: rgb(216,57,49); background-color: inherit">在作业文档里展示出 GitHub 的 Commit 记录截图，并且说明你为什么要 Commit</span>，**&#x8BC1;明你不是一次性传上去就不管了，而是懂得如何维护它。

## **三、避坑指南：给“复杂项目”的一盆冷水**

这节课云舒老师提到了一个很现实的问题：

**前端项目部署很简单（免费、快），但带数据库和后端的项目部署是几何级难度的提升。**

### 1. **如果你做的是纯静态网页（HTML/CSS/JS）**

恭喜你，GitHub + Vercel 是你的天堂，基本 5 分钟搞定，免费且稳定。这也是第五课推荐大家先做“个人主页”的原因——它容易部署，能给你建立信心。

### 2. **如果你的项目带了数据库（Supabase）和后端逻辑**

请做好心理准备，你可能会遇到（包括但不限于）以下问题：

* **环境变量（Environment Variables）丢失**：你在本地 `.env` 文件里配的 API Key，上传到 GitHub 是会被自动忽略的（为了安全）。**你必须在 Vercel 的 Settings -> Environment Variables 里，手动把你所有的 Key 再填一遍。**

  * **<span style="color: rgb(216,57,49); background-color: inherit">但是一定要记得不要把API Key发布在公开可见的仓库目录里！</span>**

* **跨域问题（CORS）**：云舒老师在答疑时也提到了，本地访问后端没问题，上了线域名变了，后端可能会拒绝服务。

* **我的建议**：如果你是小白，且没有强烈的业务需求，**不要在这个阶段死磕复杂的后端部署**。先把前端部署跑通。如果你一定要部署复杂项目，请善用 AI。

### 3. **如何用 AI 解决部署报错？**&#x90E8;署报错时的 Log（日志）通常非常晦涩。

• ❌ **错误做法**：盯着 Vercel 的报错红字发呆。

• ✅ **正确做法**：

&#x20;   1\. 在 Vercel 部署失败的页面，找到 `Build Logs`。

&#x20;   2\. 把那一大段看不懂的红色报错复制下来。

&#x20;   3\. 扔给 AI（Claude/ChatGPT）：**“我在 Vercel 部署 React 项目时遇到了这个报错 \[粘贴日志]，请告诉我这是什么原因，并给出修改代码的具体步骤。”**

&#x20;   4\. 对于复杂架构，直接问 AI：**“我的项目包含前端 \[技术栈] 和后端 \[技术栈]，请帮我设计一个最简单的免费部署方案。”**
