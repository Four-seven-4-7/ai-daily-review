## 核心认知（读这个）

### 什么是"优秀作业"

优秀作业 ≠ 最复杂的产品 ≠ 最完整的文档 ≠ 最漂亮的展示

**优秀作业 = 清晰的思维过程 + 深度的用户理解 + 严格的自我约束**

### 为什么要写优秀作业

* ❌ 不是为了得高分

* ✅ 而是为了训练自己的"产品思维"

* ✅ 而是为了学会"从问题出发"而非"从工具出发"

* ✅ 而是为了建立"用最小投入验证最大价值"的思维习惯

这套思维方式在你后续的创业、工作、生活中都能用上。

## 作业一：痛点挖掘机（3-4小时）

### 第一步：观察与记录（30分钟）

**做什么**：回顾过去一周，列出3个真实发生的"麻烦、低效、反复、不得不做"的场景。

**怎么做**：

1. 打开日历和微信记录

2. 找出那些让你"崩溃、烦躁、想吐槽"的时刻

3) 记录：时间、地点、具体做的什么、为什么烦

**好的记录长这样**：

```plain&#x20;text
场景1：周一晚8点，在家整理周报
  做什么：需要从3个不同的地方找信息（会议记录、项目表、个人笔记）
  为什么烦：每次都要反复找，很浪费时间
场景2：周三下午，整理支付截图
  做什么：3张支付截图放了一周，需要一张张手工输入到记账app
  为什么烦：重复打字特别费力，容易出错，所以一直拖延
场景3：周五，做投资决策
  做什么：被邮件、新闻、朋友建议淹没，信息太散乱
  为什么烦：没法快速判断，决策时间特别长
```

### 第二步：选最值得解决的（1小时）

**做什么**：从3个场景中选一个，进行深度分析。

**怎么选**：问自己这4个问题

* 这个问题多久发生一次？（每天、每周、每月）

* 持续多长时间了？（几周、几月、几年）

* 每次花多长时间？（5分钟、1小时、更多）

* 如果解决了，生活会改变多少？（改变价值高吗）

**优先级公式**：高频率 × 长期存在 × 高时间成本 × 高改变价值 = 最值得解决

### 第三步：深层分析（2小时）

选定一个场景，进行三层分析。

#### 第一层：现在怎么做（What）

**清晰描述现在的做法**：

```plain&#x20;text
✅ 我现在的做法：
   1. 从微信、邮件、文件夹找PDF（3-5分钟）
   2. 拖到对应项目文件夹（1分钟）
   3. 逐个重命名"项目名+日期"（5分钟）
   4. 核对有没有漏掉的（3-5分钟）
   总耗时：12-16分钟，每周做2-3次
```

#### 第二层：最想摔键盘的地方（Why烦）

**不是"不行"，而是"最消耗精力的环节"**：

```plain&#x20;text
✅ 最想摔键盘的地方：逐个重命名
   为什么烦：
   - 重复的机械操作，特别消耗脑力
   - 容易出错（名字写错或重复）
   - 一个操作出错，还要全部重做
   - 这个痛苦导致我经常拖延不做
```

#### 第三层：根本原因是什么（Root Cause）

**不是工具问题，而是什么问题？**

```plain&#x20;text
✅ 根本问题分析：
问：为什么我不想做这件事？
答：因为流程太繁琐，有太多重复的手工操作
问：如果有一个"更好看的App"，能解决吗？
答：不行，问题不在工具界面，在于机械操作本身
问：那真实问题是什么？
答："跨渠道查找" + "重复命名" + "易出错核对"
    的重复操作导致的心理负担和时间成本
根本原因：不是"我不会管理文件"
        而是"人类被机械操作消耗"
```

### 第四步：用户识别（30分钟）

**做什么**：这个问题的受害者是谁？

**不是"所有人"，而是具体的某群人**：

```plain&#x20;text
✅ 好的用户描述：
用户是谁：职场打工人（特别是项目管理人员）
特征：
- 每天需要从多个渠道收集文件（邮件、微信、网盘）
- 工作中PDF文件很多（报告、合同、资料）
- 对文件组织有要求，但手工操作费时费力
为什么有这个痛点：
- 工作流程中信息渠道分散
- 需要按项目组织，但重命名很繁琐
规模：我认识至少5个这样的人
❌ 不好的用户描述：
- "所有人都可能需要"（太宽泛）
- "想管理自己文件的人"（太模糊）
- 没有说清楚"为什么这群人有这个问题"
```

### 第五步：提炼用户真实需求（1小时，最关键）

**这一步90%的人做不好。关键是要"剥离工具形状"。**

**常见错误**：

```plain&#x20;text
❌ "我需要一个PDF管理工具"
❌ "我需要一个智能分类软件"
❌ "我需要一个能自动重命名的App"
   （这些都还在"工具思维"里）
```

**正确的做法**：

```plain&#x20;text
问：现在为什么烦？
答：因为PDF分散在多个地方，要重复手工命名
问：如果解决这个烦恼，用户能获得什么？
答：快速把PDF按项目分类，不用反复核对
问：这不是"产品"，而是什么？
答：一种"能力"——自动发现和归类文件
✅ 用户真实需求：
"不用手动翻找分散渠道的PDF，能快速把它们按项目归类好，不用反复核对"
注意：这里没有"App"、"工具"、"软件"这样的词
而是描述了"用户想要的状态改变"
```

**判断标准**：

* 用户真实需求里有"工具"这个词吗？有的话，重新写

* 这个需求能用多个方案来满足吗？（而不是只有一个工具）

* 用户会为了这个需求改变行为吗？

## 作业二：MVP架构师（4-5小时）

### 第一步：设计核心链路（1小时）

**最重要的原则**：MVP只有一个核心链路，不能"多一个都算违规"。

**做什么**：基于作业一的"用户真实需求"，设计最小的验证装置。

**思考过程**：

```plain&#x20;text
用户真实需求：不用手动翻找，快速按项目自动归类PDF
  ↓
要验证这个需求，最少需要什么？
  ↓
答：自动发现PDF + 识别项目 + 自动归类
  ↓
这个核心链路成功的标志：用户不用手工操作，就看到PDF被分好类了
  ↓
不需要做什么：搜索、标签、权限管理、分享功能等
```

**好的核心链路长这样**：

```plain&#x20;text
✅ 记账MVP：
   支付截图 → 自动识别 → 显示结构化记录
   （只做这个，其他都不做）
✅ 读书会MVP：
   发起人发起共读 → 读者报名 → 读完评价
   （用邮件提醒，讨论用微信/Zoom）
✅ PDF管理MVP：
   扫描文件夹 → 识别PDF来源 → 按项目自动归类
   （不做搜索、分享、权限管理）
❌ 坏的核心链路：
   - 记录 + 分类 + 统计 + 预算提醒（太多了）
   - "一个文件管理系统"（这不是链路，这是产品描述）
```

**检查清单**：

* [ ] 这个链路能用一句话说清楚吗？

* [ ] 如果只做这一个功能，能验证"用户真的需要吗"吗？

* [ ] 有没有我可以删除的环节？

* [ ] 我是否在完美主义驱使下加入了不必要的功能？

### 第二步：选择乐高组件（1.5小时）

**做什么**：为每个需要的"能力"选择合适的工具，并说出"为什么"。

**乐高组件清单对照**：

```plain&#x20;text
能力1：做用户界面（前端）
  选项：Enter.pro / Lovable / NextJS等
  问：我选谁？为什么？
  答：选Enter.pro，因为我只是想快速做出可用的表单和页面，不想自己写代码
能力2：存储数据
  选项：Supabase / Firebase / 其他
  问：我选谁？为什么？
  答：选Supabase，因为我需要关系型数据库（表之间有联系）+ 内置Auth系统
能力3：自动识别PDF
  选项：Claude API / GPT-4V / 其他AI
  问：我选谁？为什么？
  答：选Claude API，因为Claude在识别图片和文件方面很强
能力4：用户登录
  选项：Supabase Auth / Auth0 / 其他
  问：我选谁？为什么？
  答：用Supabase自带的Auth，不想新增组件
能力5：发邮件提醒
  选项：Resend / SendGrid / 其他
  问：我选谁？为什么？
  答：选Resend，因为API简洁，集成快
能力6：上线展示
  选项：Vercel / Netlify / 其他
  问：我选谁？为什么？
  答：选Vercel，因为配合NextJS最方便，自动部署
```

**最关键的一点**：为每个选择都要说出"为什么"。

```plain&#x20;text
❌ 不好的做法：用Supabase、Vercel、Claude API
✅ 好的做法：
   用Supabase——因为需要关系型数据库+Auth
   用Vercel——因为要快速部署，Vercel配合Next.js最方便
   用Claude API——因为识别能力强，调用简单
```

### 第三步：预见用户不买账的原因（1.5小时）

**这是最能体现思考深度的部分。**

**做什么**：反向思考"为什么用户不用这个产品"。

**思考维度**：

1. **心理层面**——用户为什么不想用

```plain&#x20;text
问题：用户的心理障碍是什么？
例子：
- "我担心数据被泄露" → 对策：强调本地处理，告诉用户数据怎么保护
- "这个似乎没啥用，手工命名也快" → 对策：显示"你能节省多少时间"
- "我不信任陌生软件" → 对策：展示真实用户的评价和使用案例
```

2. **行为习惯层面**——用户的操作障碍

```plain&#x20;text
问题：用户的真实习惯是什么？我的产品有违背吗？
例子：
- "我的用户都是50岁的HR，不熟悉新工具"
  对策：简化界面，提供视频教程，或者做成浏览器插件
- "我的用户习惯用本地文件夹，不想上云"
  对策：支持本地存储，可选上云同步
```

3. **环境因素层面**——客观障碍

```plain&#x20;text
问题：什么外部因素会导致"用户没法用"？
例子：
- "公司IT禁用某些工具" → 对策：提供替代方案
- "用户在没网的地方无法使用" → 对策：离线模式
- "跨时区问题导致提醒时间不对" → 对策：让用户自设时区
```

4. **工程层面**——产品设计问题

```plain&#x20;text
问题：我的MVP有没有技术门槛？
例子：
- "我计划做6个月才能上线" → 问题：太慢了，需要简化MVP
- "我依赖太多第三方服务" → 问题：有一个挂了整个系统就瘫痪
```

**具体的避坑指南写法**：

```plain&#x20;text
【避坑1：用户不信任自动化】
症状：
用户看到"自动识别文件"，会想"机器会不会乱分类？"
然后继续用手工命名，因为"至少我知道没错"
根因：
用户对"陌生的自动化工具"有心理抵触
特别是涉及"文件管理"这种重要的事
对策：
① 先让用户看到"自动分类"的结果，再给选择"确认"的机会
② 显示"识别置信度"（80% 有把握）
③ 提供"手工编辑"功能，用户可以改错了的分类
④ 展示"其他用户修正的比例"（99.2% 自动分类是对的）
```

## 作业呈现（核心建议）

### 怎么写得清晰

**每一个观点都要有"为什么"**：

```plain&#x20;text
❌ "我选择Supabase"
✅ "我选择Supabase，因为我需要Auth系统和关系型数据库"
❌ "用户的痛点是效率低"
✅ "用户的痛点是机械性的重复操作（拖文件→重命名→核对）导致长期拖延"
```

**用具体的数字和例子**：

```plain&#x20;text
❌ "花很多时间"
✅ "每周3次、每次12-16分钟，年耗时超过10小时"
❌ "很多人需要"
✅ "我认识的5个项目管理人员都有这个问题"
```

**呈现结构要清晰**：

* 先说"是什么"（现象）

* 再说"为什么"（根因）

* 最后说"怎么办"（解决方案）

### 怎么展示有说服力

```plain&#x20;text
❌ 直接给答案
✅ 展示思考过程
❌ "我觉得应该这样做"
✅ "我问了3个用户，他们都说最烦的地方是..."
❌ "这是最好的方案"
✅ "对比了X、Y、Z方案，选择这个因为..."
```

## 最后的话

### 为什么优秀作业很重要

写一份优秀作业不只是得高分，而是：

1. **训练你的"产品思维"** - 从问题出发，而非工具出发

2. **建立"MVP精神"** - 用最小投入验证最大价值

3) **学会预见问题** - 从用户角度反向思考

4) **记录你的成长** - 这份作业是你成长的见证

### 如果你遇到困难

```plain&#x20;text
问题：痛点不够深
→ 解决：问3个朋友"你也有这个问题吗"，听听他们怎么解决的
问题：用户太模糊
→ 解决：不要说"所有人"，而是"描述一个真实的人"
问题：MVP太复杂
→ 解决：删掉最不重要的那个功能，再删一个
问题：避坑太笼统
→ 解决：想象一个真实的用户，他为什么不用你的产品
问题：不知道怎么呈现
→ 解决：找一个优秀作业，看看他怎么写的，参考结构（不要抄内容）
```

### 关键的思维转变

```plain&#x20;text
从：我想做一个什么样的产品
到：我想解决什么样的问题
从：用Entry.pro、Supabase、Vercel来做
到：这个问题为什么需要这些工具
从：完成一次作业任务
到：通过作业训练自己的思维方式
这三个转变，就是优秀作业和普通作业的本质区别。
```
